## Brainstorm

Task #25: orchestrate SCIM user creation end-to-end. Receives a SCIM user body from router, acquires idempotency lock, creates user in Brivo, writes idmap, releases lock. Returns `(BrivoUser, scim_id)` ‚Äî router does SCIM mapping.

Scope: `app/services/create_user.py`. Three steps via `run_saga`: lock ‚Üí POST Brivo ‚Üí write idmap + DEL lock.

Constraints:
- Lock key: `lock:brivo:create:user:{external_id}` (300s TTL via `store.acquire_lock`) ‚Äî 409 if already held (concurrent create in flight)
- `scim_id` = UUID v4, generated before saga starts; stable if saga retried by IdP
- Step outputs shared via closure (mutable dict between steps)
- Rollback: step 1 ‚Üí release lock; step 2 ‚Üí delete Brivo user; step 3 ‚Üí del idmap (lock already released in forward)
- `external_id` sourced from SCIM `externalId` field

Related: [Saga Base Runner](20260620163423_saga_base_runner.md) [Field Mapper Write](20260620114246_field_mapper_write.md)

## Story

As SCIM users router, want create-user saga, so POST /Users atomically provisions Brivo user and idmap without duplicate risk.

AC:
1. `create_user(body: ScimUserRequest, store: RedisStore, client: BrivoClient) -> tuple[BrivoUser, str]` ‚Äî async; returns `(brivo_user, scim_id)`
2. `external_id = body.externalId or body.userName` ‚Äî `externalId` is optional (SCIM spec); `userName` is the fallback lock key and idmap identifier
3. Generates `scim_id` = UUID v4; acquires lock via `store.acquire_lock("user", external_id, scim_id)` ‚Äî raises `ScimConflict` (409) if already held; lock acquired before saga starts
4. Builds `BrivoUserWrite` via `scim_user_to_brivo(body)` before saga starts
5. Step 1 ‚Äî "create-brivo-user": forward = `client.create_user(brivo_write)`, store result in closure; rollback = `client.delete_user(brivo_user.id)` (swallow 404) + `store.release_lock`
6. Step 2 ‚Äî "write-idmap": forward = `store.set_idmap(...)` then `store.release_lock`; rollback = `store.del_idmap(...)`
7. `SagaError` propagated to caller (router maps to 500)
8. Test: happy path with externalId ‚Äî lock acquired on externalId, idmap written with externalId, lock released
9. Test: happy path without externalId ‚Äî lock acquired on userName, idmap written with userName, lock released
10. Test: lock conflict ‚Üí `ScimConflict` raised, saga never starts
11. Test: Brivo create fails ‚Üí lock released (step 1 rollback), `SagaError` raised
12. Test: idmap write fails ‚Üí Brivo user deleted + lock released (step 1 rollback), `SagaError` raised

## Design

### Flow

```mermaid
flowchart LR
    A([body]) --> B[external_id = externalId or userName]
    B --> D[build BrivoUserWrite\ngenerate scim_id\nacquire lock]
    D -- lock held --> E([ScimConflict 409])
    D -- lock acquired --> F[run_saga\nstep1: create Brivo user\nstep2: write idmap + release lock]
    F -- success --> G([return BrivoUser scim_id])
    F -- SagaError --> H([propagate to router])
```

## Summary

`create_user` computes `external_id = externalId or userName` (SCIM spec makes externalId optional; userName is always present and unique), acquires idempotency lock on that key (ScimConflict on conflict), then runs a 2-step saga: create Brivo user, write idmap + release lock. Step 1 rollback deletes the Brivo user (swallow 404) and releases the lock ‚Äî `run_saga` calls the failed step's rollback too, so lock cleanup is always owned by the saga, not by the caller. Step 2 rollback deletes idmap only.
